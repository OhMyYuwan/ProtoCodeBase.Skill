ProtoCodeBase

不要让你的 Agent 重复造轮子。

让它少读一点，少猜一点，少在上下文里迷路。ProtoCodeBase 的重点不是让 Agent 写更多代码， 而是更快找到值得参考的那一小块代码，以及真正需要修改的那一小块代码。

Why It Matters

Agent 最大的浪费，不是写错几行代码

01 反复读仓库

每次都重新扫描目录、猜模块边界，真正需要改的地方反而更晚出现。

02 聊天记忆脆弱

需求、计划和取舍散落在对话里，下一次接手很难恢复工程上下文。

03 整库塞进上下文

Agent 不应该每次都把整个仓库灌进上下文，而应该按项目说明书直达目标位置。

Need First

需求先于代码

代码不是目的，需求才是。

一个项目里真正值得保存的，不只是文件本身，还有需求如何被理解、如何被拆解、如何被计划、如何被执行。 你的 Intent 仍然属于你。协议不需要把它伪装成另一种对象。 ProtoCodeBase 持久化的是程序规范化后的工程链路。

Input Intent

来自用户的真实需求

Normalized Request

可计划、可执行的项目请求

Output Trace

留给项目的工程记忆

Engineering Memory

把聊天变成工程链路

00 Intent 用户的需求本身 01 Request 定义语义目标 02 Plan 限定执行范围 03 Change 记录真实变更

软件修改不再只是散落在聊天窗口里的脆弱记忆，而是可以被下一次 Agent 接住的工程上下文。

Token Economy

省 token，不是省理解

Agent 最贵的浪费，不是多写几行代码，而是反复走进同一片森林， 却找不到你要的那棵树。

如果一个成熟项目是一台完整的机器，就不要让 Agent 每次都从拆零件开始。 给它一份说明书：哪里能改、哪里不能动、改了会影响什么。

Capabilities Project Map Route
01 Capabilities

描述这个 App 实际能做什么，以及每个能力的边界。

02 Project Map

描述项目如何组织，哪些模块承载哪些责任。

03 Route

帮助 Agent 从需求直接定位到目标代码、目标组件和目标服务。

这样，Agent 不必把整个仓库塞进上下文；它可以释放旧上下文，只保留被规范化的工作信息， 下一次回来也能从说明书里重新找到目标位置。

Industrial Templates

与其创造，不如继承

成熟软件已经走过的路，不必每次从空白文件重新发明。

文件管理、权限、协作、编辑器、工作流、历史记录、错误处理、部署方式和用户界面， 都可以沉淀成 Agent 可参考、可裁剪、可合并的说明书。

Repository Package

Skill Wrapper

让 Agent 识别 ACP v1.0.0 项目，并按协议进入工作流。

Protocol References

包含公共草案、Agent playbook、审计追踪和版本说明。

Authoring Templates

提供 Request、Plan、Change 的对象模板。

Registry & Docs

提供 skill registry、GitHub Pages、README、许可证和使用说明。

Project Declaration

项目如何声明自己使用 ACP

protocol: acp
protocol_version: "1.0.0"
required_skill: acp-v1-0-0
技能选择

仓库声明协议版本，Agent 读取对应 Skill，而不是猜测项目规则。

版本边界

未来的协议版本可以独立添加，避免不同规则混在一起。

权利说明

Skill wrapper 采用 MIT；内置协议文本和 reference package 可能有自己的 LICENSE。

Install

把协议交给Agent

安装后，目标仓库声明自己的协议版本；Agent 读取对应 Skill， 再按照协议说明书进入项目，而不是每次重新拆解整个系统。

Terminal 
npx skills add https://github.com/OhMyYuwan/ProtoCodeBase.Skill.git --skill acp-v1-0-0
